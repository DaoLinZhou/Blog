TCP的滑动窗口可以对流量进行控制, 来避免拥塞的发生, 本文会讲述一下滑动窗口的大致原理

RTT 和 RTO

RTT : 发送一个数据包到收到对应的ACK, 所花费的时间

RTO : 重传时间间隔, 如果收到ACK 就不会重传, RTO 是通过 RTT 计算出来的


TCP 的滑动窗口

首先我们知道 TCP 会把数据拆分成段进行发送.

但是不可能一段一段地进行发送, 等到上一段数据被确认之后才发送下一段数据, 所以必须批量传输, 并且 TCP 要解决可靠传输, 包乱序的问题, 所以TCP要知道网络实际处理数据的带宽, 或是实际处理速度, 这样才不会引起网络拥塞而导致丢包.


TCP使用滑动窗口做流量控制与乱序重排

  • 保证TCP的可靠性
  • 保证TCP的流控特性

TCP的报文格式中有window, 即:窗口

它的作用是: 接受方通知发送方自己还有多少缓冲区可以接受数据

而发送方根据接收方的处理能力而发送数据, 不对导致接受方处理不过来


窗口数据的计算过程

左边是TCP协议发送端的缓冲区

右边是接收端缓冲区

左边往右边发数据


左边发送方:

​ 对于发送方 LastByteAcked 指向收到的连续最大ACK的位置. 也就是目前已经收到ACK到这个位置. LastByteSent 指向已发送的最后一个字节的位置, 该位置只是发出去了, 但还没有收到ACK. LastByteWritten 指向上层应用以写完的最后一个位置, 即: 当前程序, 已经准备写好需要发送的最新的数据.

​ LastByteAcked 之前: 发送并收到ACK

​ LastByteAcked - LastByteSent : 发送, 但没有接受到ACK

​ LastByteSent - LastByteWritten : 待发送的数据

​ LastByteAcked 到 LastByteWritten 之间是连续的, 没有间隔的


右边接收方:

LastByteRead 指向上层应用已经读完的最后一个字节的位置

NextByteExpected 指向收到的连续最大的sequence的位置

LastByteRcvd 指向已收到最后一个字节的位置

NextByteExpected 和 LastByteRcvd 之间有一些 seq 并没有到达(空白)


计算EffectiveWindow

AdvertisedWindow 我能够接受的窗口(接收方还能处理数据的量)

MaxRcvBuffer 接收方能接受的最大数据量(接收端缓存时的大小)

(LastByteRcvd - LastByteRead) 是为已收到的数据和未收到的数据留出的空间

AdvertisedWindow = MaxRcvBuffer - (LastByteRcvd - LastByteRead)

之后把 AdvertisedWindow 发送(告知) 给 发送端.

发送端要保证LastByteAcked - LastByteSent中间的数据量要小于AdvertisedWindow . 即: 已发送但没响应的数据量要小于接收方还能处理数据的量


LastByteSent - LastByteAcked 为已发送的数据量

AdvertisedWindow 接收方还能处理数据的量

EffectiveWindow 还能发送的数据量

EffectiveWindow = AdvertisedWindow  - (LastByteSent - LastByteAcked)

TCP会话发送方

TCP会话发送方, 在任何时候, 缓存都可以分为4类

Category #1 : 已经发送并且得到端的回应的, 即: LastByteAcked 之前

Category #2 : 已经发送但还没收到端的回应的, 即: LastByteAcked - LastByteSent

Category #3 : 未发送但对端允许发送的, 即: LastByteSent - LastByteWritten

Category #4 : 未发送, 且由于达到window的大小不允许发送的数据


而Category #2, #3 所组成的连续空间, 则被称为发送窗口

窗口滑动

当收到接收方新的ACK(对于后续窗口字节的确认)时, 窗口进行滑动

如图所示, 如果32, 33 没被确认, 但34 被确认了, 窗口也不会向右滑动

只有当连续被确认之后(32, 33, 34 …), 窗口才会滑动, 而没滑动以前, 52 即之后的数据是不能被发送的.

假设到35 都被确认了, 那么窗户滑动4位, 变成从36开始, 进而就能发送52 - 55 的数据了


TCP会话接收方

对于接收方, 有三种状态:

Category #1 + 2 : 已接收, 并且已经发送回执

Category #3 : 未接收, 但可以接收

Category #4 : 未接收, 但由于达到窗口的阈值, 所以不能接收


默认没有应用延迟, 不存在已接收但未回复的状态

其中 Category #3 (未接收, 但可以接收) 就是接收窗口

接收方的滑动窗口和发送方的一致, 只有收到连续的ack才会移动左边界,

如果前面没接收, 而后面接收, 则不会滑动窗口, 确保数据会被重传.


注意:

滑动窗口的大小可以根据一定策略动态调整, 应用会根据自身处理能力的变化通过本段TCP窗口的大小的控制, 来实现对端的发送窗口进行流量控制
