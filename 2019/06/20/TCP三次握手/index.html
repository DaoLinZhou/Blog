我将通过Wireshark 这款软件对B站进行抓包, 来分析TCP的三次握手

TCP 的三次握手

当应用程序希望通过TCP与另一个应该程序通信时, 则会发送一个通信请求, 这个请求必须被送到一个确切的地址.

在双方握手之后, TCP 将在两个应用之间建立一个全双工（Full Duplex）的通信. 这将占用两个计算机之间的通信线路. 直到它被一方 或 双方关闭为止


什么是全双工 (Full Duplex)

计算机 A 给 B 发送信息, 在发送信息的同时, B 也能给 A 回发信息


流程图

可以看到 服务器向客户端发送ack 就是 SYN 和 SEQ 的和 (x+1), 同时自己也要初始化一个序列号

而 客户端 向服务器发送的第二次请求中, seq就等于上次收到的ack的值, 并且携带的ack为y+1, 即期望下一次服务器发送的seq


抓包

准备工作: 获取B站的ip地址

既然要抓包, 首先要知道要抓 “谁” 的包, 通过 ping 命令获取 B站 ip

正式开始

启动 Wireshark 选择WALN, 并且开启捕获 刷新b站页面

设置过滤规则:

​ 我们只关注 ip 为 148.153.38.118 的相应和请求

ip.src == 148.153.38.118 || ip.dst == 148.153.38.118

结果

148.153.38.118 是b站的 ip, 那么本机的ip便为 172.29.154.49

解析

下面的解析过程最好配合流程图观看

首先本机向B站发送一个SYN包, seq (从本机往B站的seq) =0,

之后b站回复本机一个(SYN,ACK)包 ack = 0+1 = 1, seq (从B站往本机的seq) = 0

最后本机给B站回复一个ACK包 seq = (b站给我们的ack) = 1, ack = seq + SYN = 0 + 1 = 1;

上面这些步骤就是TCP的三次握手


为什么要三次握手?

为了初始化Sequence Number 的初始值, 双方要通知对方自己的secquence Number.

这个号会作为以后通讯的序号, 以保证收到的数据不会因为网络传输问题而乱序 (TCP 会用这个序号来拼接数据)

最后一次本机发往B站的ACK包, 是告诉服务器”本机收到你的seq number了”


从一个故事说起

我记得自己以前看过一个故事, 描述的就是TCP为什么要握手3次, 但是如今已经找不到原文了, 所以只能把我记忆中的故事大概讲述一遍.

故事

现在有三个国家, A B C.

A要联合B一起攻打C, 于是送信给B国, 告诉他XXX时间, 你我两国一起攻打C国.

但问题是, 信使送信可能会被C国截获, 导致B不知道要攻打C国, 而A国却在约定的时间傻fufu地冲上去了.

所以B国要给A国回个信, 告诉他, “我知道了, 在XXX时间, 咱俩一起进攻C国”

问题又来了, 信使送信可能会被C国截获, B国知道要在XXX时间攻打C国, 但A不知道B知道(有点绕口),

A 没收到信件, A就怀疑, 是不是自己上次发的信被截获了, 而B不知道有这么回事. 所以与其傻fufu的攻打C国, A会按兵不动. 而B又傻fufu地攻打C国去了.

此时A如果收到信件, 要给B回信: “我知道你知道了, 到时候一起去干他”

如果这么持续下去, A 与 B要无限地给对方发送信件. 于是, 人们(计算机科学家们)认为, 成功传输3次, 就可以证明互相之间是可以通信的


首次握手的隐患

问题的起因

Server 收到 Client 的SYN, 回复SYN-ACK 没有收到ACK确认

此时连接处于中间状态, 没有成功, 也没有失败

Server 不断重试直到超时, Linux 默认重试5次, 重试间隔从1秒开始, 每次翻倍, (1, 2, 4, 8, 16) 在第五次发出去之后, 会等待32 秒, 一共会等待63秒才判定为超时,断开连接


因此可能会有SYN Flood攻击的风险

恶意客户端会给服务器发送一个SYN报文, 之后他就下线了,

而服务器需要默认等 63 秒才会断开连接. 攻击者重复多次, 便把 SYN 连接队列耗尽, 正常请求无法处理


防范措施

SYN 队列满后, 通过tcp_syncookies参数回发SYN cookies

通过 原地址端口, 目标地址端口, 时间戳 打造出一个特别的seq number 简称SYN cookie.

如果是攻击者, 由于它下线了, 所以不会有任何反应.

而正常连接则会把SYN cookies 发回来, 服务端通过SYN cookie建立连接

建立连接后, Client出现故障怎么办

保活机制

在一段时间(保活时间 keep alive time) 连接处于非活动状态,

开启保活的一端将向对方发送保活探测报文, 如果未收到相应则继续发送

直到尝试次数达到保活探测数仍未收到相应则中断连接, 认为对方主机为不可达
