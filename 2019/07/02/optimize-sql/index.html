如何定位并优化慢查询SQL

开放性问题, 只有大致思路

  1. 通过慢日志定位慢查询sql
  2. 使用explain等工具分析sql
  3. 修改sql或者尽量让sql走索引

查看变量

show variables like "%quer%";

slow_query_log : 是否开启慢日志

slow_query_log_file : 慢日志记录位置

long_query_time : 时限, 上图: 一次如果超过10秒就会被记录(通常超过1秒就是慢sql)


显示慢查询数量

show status like "%slow_queries%";

explain 的使用

一般explain放在select查询语句的前面, 用于描述mysql如何执行查询操作, 以及mysql成功返回结果集所要执行的函数


关键字段:

type:

如果type是最后两个,代表本次走的是全表扫描, 是需要优化的语句


extra:


修改sql, 让它走索引

例:

select name frm person_info_large order by name desc;

可以看到, name没有索引, 而account有索引. 如果用account 替代name就可以让sql走索引

select account frm person_info_large order by account desc;

或者直接为name加上索引

alter table person_info_large add index idx_name(name);

还可以通过force index() 测试哪个索引是比较好的


联合索引的最左匹配原则的成因

联合索引: 由多列组成的索引, 上图中这句话就是联合索引

KEY `index_area_title` (`area`, `title`)

最左匹配原则:

​ 假设有A,B 两列, 将A , B 都设置为索引(联合索引), 顺序是A,B.

​ 在where 语句中 

// 走索引
where A = xx and B = xxx
where A = xx
// 不走索引
where B = xxx

定义:

​ 1. 最左前缀匹配原则, 非常重要的原则, mysql会一直向右匹配直到遇到范围查询 (>, <, between, like) 就停止匹配, 比如

where a=3 and b=4 and c>5 and d=6

如果建立(a,b,c,d) 顺序的索引, d是用不到索引的,

如果建立(a,b,d,c)顺序的索引, 则都可以用到.

a, b, d的顺序可以任意调整.


  1. = 和 in 可以乱序, 比如
where a=1 and b=2 and c=3

建立(a,b,c)索引可以任意顺序, mysql的查询优化器会帮你优化成索引可以识别的形式


解答

mysql 创建复合索引的规则是会首先对复合索引最左边, 也就是第一

个索引字段的数据进行排序, 之后再对第二个字段进行排序, 所以第一个字段是绝对有序的, 而第二个字段则是无序的. 因此直接使用第二个字段进行条件判断是用不到索引的


索引是建立越多越好吗

物极必反, 肯定不是建立越多越好

  1. 数据量小的时候不需要建立索引, 建立会增加额外的索引开销
  2. 数据变更需要维护索引, 因此更多的索引意味着更多的维护成本
  3. 更多的索引需要更多的空间
