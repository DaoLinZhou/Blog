Hash Map

哈希表, 又称 hash map 或 hash table

map这类数据结构存放的是key-value这样的键值对

java 中 treemap 就是用红黑树实现的map. 用于存放键值对.

而hash map则是用空间换时间.

通过哈希函数 计算hash值, 将键转换为索引.

treemap可以看作是一个特殊的hash map. 也就是哈希函数转化key后的hash值都是一样的. 所有的key-value存在一个红黑树中.

而更一般的hash map则是把相同hash值的key-value存入一个红黑树中.

这样均摊下来每个树的高度就变小了, 我们拥有更多的树.


然而很难保证每一个键通过哈希函数转换后的索引都是唯一的.

这就是哈希冲突, 解决hash冲突就是哈希表中一个重要的操作.



哈希函数的设计

“键”通过哈希函数得到的”索引”分布越均匀越好.

由于存储的数据五花八门, 实际上实现起来是很难的.

对于一些特殊领域. 有特殊领域的哈希函数设计方式. 甚至有专门的论文.


整型

后续会看到, 可以把所有类型的哈希函数都转化成整型的哈希函数.

  1. 小范围的正整数可以直接使用. 直接用这个数值作为索引即可
  2. 小范围负整数进行偏移, 成为正整数
  3. 对于大整数, 例如身份证号. 最常用的做法就是取模.

然而对几进行取模非常重要, 如果这个数选择不好就极有可能发生分布不均匀的情况

一个简单的解决方法就是: 模一个素数. 尤其是在数据具有一定规律的时候


那么模的这个素数到底怎么取?

浮点型

可以把浮点型的哈希函数转化成整型进行处理


字符串

字符串可以看作时B进制的整型

而计算B的k次方的过程是比较慢的, 所以可以把式子转换成下面几个样子

此时写程序就会非常容易

int hash = 0;
for(int i = 0; i < s.length(); i++)
    hash = (hash * B + s.charAt(i)) % M;

复合类型

和字符串一样. 其实字符串可以理解为由多个字符组成的复合类型

假设复合类型包含年, 月, 日


转成整型处理并不是唯一的方法:

原则:

  1. 一致性: 如果a==b, 则hash(a) == hash(b)
  2. 高效性: 计算高效简便
  3. 均匀性: 哈希值均匀分布


在java中, 所有类的父类Object有一个hashCode方法, 用来获取当前类的hash值.

这个值是一个整数(可能为负数). 具体这个整型如何和数组中的索引再取对应, 这一点由hash表内部进行完成.

Object默认是根据对象的存储地址计算hashCode. 也就是说即使属性相同, 不同的对象的hashCode也可能不同.

因此如果在java中使用HashSet, HashMap. 对象如果不重载hashCode, 那么可能相同的key会插入多次. 因为即使key相同, 它们存储地址不同, hashcode就不同. 所以一定要重载hashCode方法

而hashCode只是用于帮助计算hash值, 当产生hash冲突时, 需要通过equals来判断key是否相等



哈希冲突的处理: 链地址法Seperate Chaining

哈希表的本质就是数组.

不过hashCode的结果有可能为负数所以我们需要把负号去掉

通常是取绝对值. 不过有的时候会有这样的写法

hashCode(k1) % M;    // 结果可能为负数
(hashCode(k1) & 0x7fffffff) % M;    // 0x7fffffff 的二进制是0后面跟31个1. 就是舍去符号位

如果产生哈希冲突, 则以链表的方式继续存储

其实也不一定是链表, 只要是查找表即可. 红黑树是比较常用的结构

java中最开始就是用链表. 但当hash冲突到一定程度时(同时key是可比较的时候), 就会转为红黑树.


public class HashTable<K extends Comparable<K>, V> {

    private final int[] capacity
            = {53, 97, 193, 389, 769, 1543, 3079, 6151, 12289, 24593,
            49157, 98317, 196613, 393241, 786433, 1572869, 3145739, 6291469,
            12582917, 25165843, 50331653, 100663319, 201326611, 402653189, 805306457, 1610612741};

    private static final int UPPER_TOL = 10;
    private static final int LOWER_TOL = 2;
    private int capacityIndex = 0;

    private TreeMap<K,V>[] hashtable;
    private int M;
    private int size;

    public HashTable() {
        this.M = capacity[capacityIndex];
        size = 0;
        hashtable = new TreeMap[M];
        for (int i = 0; i < M; i++){
            hashtable[i] = new TreeMap<>();
        }
    }

    private int hash(K key){
        return (key.hashCode() & 0x7fffffff) % M;
    }

    public int getSize(){
        return size;
    }

    public void add(K key, V value){
        TreeMap<K, V> map = hashtable[hash(key)];
        if(map.containsKey(key))
            map.put(key, value);
        else {
            map.put(key, value);
            size++;

            if(size >= UPPER_TOL * M && capacityIndex + 1 < capacity.length)
                resize(capacity[++ capacityIndex]);
        }
    }

    public V remove(K key){
        TreeMap<K, V> map = hashtable[hash(key)];
        V ret = null;
        if(map.containsKey(key)) {
            ret = map.remove(key);
            size --;
            if(size < LOWER_TOL * M && capacityIndex - 1 >=0)
                resize(capacity[-- capacityIndex]);
        }
        return ret;
    }

    public void set(K key, V value){
        TreeMap<K, V> map = hashtable[hash(key)];
        if(!map.containsKey(key))
            throw new IllegalArgumentException(key + " doesn't exist");
        map.put(key, value);
    }

    public boolean contains(K key){
        return hashtable[hash(key)].containsKey(key);
    }

    public V get(K key){
        return hashtable[hash(key)].get(key);
    }

    // 当 size 不在 [LOWER_TOL*M, UPPER_TOL*M) 这个区间时进行 resize
    private void resize(int newM){
        TreeMap<K, V>[] newHashTable = new TreeMap[newM];
        for (int i = 0; i < newM; i++)
            newHashTable[i] = new TreeMap<>();

        int oldM = this.M;
        this.M = newM;

        for (int i = 0; i < oldM; i++){
            TreeMap<K, V> map = hashtable[i];
            for (K key: map.keySet()){
                newHashTable[hash(key)].put(key, map.get(key));
            }
        }
        this.hashtable = newHashTable;
    }

}


开发地址法

和链地址法齐名的另外一个哈希冲突的处理方式就是开发地址法.

它的思想就是当产生哈希冲突时, 继续向后查找时候有空余位置.

例如此时插入31. 而我们假设31的hash值为1. 而1这个位置有元素了. 那么31就继续向后看有没有空位置.

因此31插入到位置2

再插入81, 假设81的hash值也为1. 此时位置1, 2都被占用了. 那么81就插入到位置3

也就是遇到哈希冲突就+1.


基于此也有各种各样的改进方法.

例如平方探测

遇到哈希冲突 +1, +4, +9, +16

平方探测也有一定规律性.

遇到哈希冲突时, 使用另外一个hash函数 +hash2(key)

使用开放地址法, 这个数组空间是有可能被占满的



C++中unordered_map使用自定义key

c++的unordered_map就是hashMap.

然而对于我们自定义的结构来说, c++不知道如何计算hash值. 因此和java一样. 我们要重载两个方法, 一个用于比较相等, 一个用于得到hash值


重载比较函数比较简单, 函数必须是const的

struct Employee{
    string name;
    int age;
    bool male;

    Employee(string name, int age, bool male):name(name),age(age), male(male){};

    bool operator==(const Employee& e) const {
        return name == e.name && age == e.age && male == e.male;
    }
};

而重载计算hash的函数则需要另写一个类. 这个类重载()运算符, 也需要是const函数

struct MyHashFunc {
    std::size_t operator()(const Employee& e) const {
        return e.age*10 + e.male;
    }
};

使用的时候, unordered map的template中传3个类

int main() {
    unordered_map<Employee, int, MyHashFunc> map;
    map[Employee("aaa", 11, true)] = 11;
    cout << map[Employee("aaa", 11, true)] << endl;
    return 0;
}


更规范一些的写法是:

struct Employee{
    string name;
    int age;
    bool male;
    Employee(string name, int age, bool male):name(name),age(age), male(male){};
    bool operator==(const Employee& e) const {
        return name == e.name && age == e.age && male == e.male;
    }
};

namespace std {
    template<>
    struct hash<Employee> {
        std::size_t operator()(const Employee &e) const {
            return hash<int>()(e.age) + hash<int>()(e.male);
        }
    };
}


int main() {
    unordered_map<Employee, int, std::hash<Employee>> map;
    map[Employee("aaa", 11, true)] = 11;
    cout << map[Employee("aaa", 11, true)] << endl;
    return 0;
}


思考:

都说hash table牺牲了顺序性.

然而仔细想一想, hash table相当于把一个树结构拆分成多个树结构. 每个树中的key的hash都是一样的(hash冲突). 每个树都是有序的. 当我们要顺序输出时. 完全可以使用堆/优先队列来merge多个树.

那么时间复杂度就是O(nlogm), n是hash table中数据的个数. 有m个tree.这个时间复杂度并不比O(n)高太多. 因为通常m是远远小于n的.
