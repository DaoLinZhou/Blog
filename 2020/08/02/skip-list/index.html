跳跃表

skip list 是对链表的扩展

普通的有序链表的增删改查的时间复杂度都是O(n)

而跳跃表的增删改查的时间复杂度都是O(logn)

跳跃表设计的初衷是为了取代平衡树(例如红黑树)

跳跃表在 Redis 和 LevelDB 中都有应用


思路

这是一个普通的有序链表

如果我们进行查找, 需要由左到右对每个节点进行访问. 然而我们知道这个链表是有序的. 那么我们是否可以让它可以跳过一些元素?

先从上面的链表找, 直到找到的元素大于寻找到元素, 就降低一层继续查找.

绿色线就是寻找路线


更进一步:



跳跃表的搜索

正如上面所述:

  1. 从顶层链表的的首元素开始, 从左往右搜索, 直到找到一个大于等于目标的元素, 或者空节点.
  2. 如果元素等于目标元素, 则表明被找到了.
  3. 如果该元素大于目标元素或着已经到达链表尾部, 则退回到当前层的前一个元素, 转入下一层继续搜索


跳跃表的添加

跳跃表添加元素的层数是随机的 (服从几何分布)

1/2 的概率新添加的元素的层数为1, 1/4的概率层数为2, 1/8的概率层数为3…


从 0, 1 中随机选一个数, 如果为1, 则添加一层, 如果为0则停止添加.

假设有32个元素. 理想情况下:

第一层有32个元素. 每个元素层数至少为1

第二层有16个元素, 1/2的概率层数不为1, 就是说二层及以上层数的元素为16

第三层有8个元素, 3/4的概率层数小于3, 1/4的概率层数为3层即以上

… …

自然时间复杂度就是O(logn)


其实找到节点的位置就可以进行增删了.

然而要注意: 添加元素或删除元素都有可能会导致层数的变化.


而且这个概率是可以指定的. 通常使用 1/2 或者 1/4



使用数组的实现

代码实现

这种实现方式的思想就是使用数组来存放指向next的指针.

这种方式的好处就是占用空间小, 每个元素只用一个节点.


使用多节点实现

然而还有另外一种实现方式. 就是每个元素使用多个节点

例如这张图, 元素1就有3个节点, 每个节点有一个指向右边的指针和一个指向下方的指针.

更进一步, 如果要实现双向的, 上下左右各需要一个指针.

而且每个元素的要使用多个节点. 这样占用的内存是较大的.



性能

根据我的测试, 100万数据规模下, 跳跃表的性能和红黑树不相上下.

如果跳跃表的 p 设计的好, 那么在一定规模下(5000万)甚至会比红黑树要快.
