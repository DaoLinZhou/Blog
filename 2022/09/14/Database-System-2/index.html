数据库实现

数据库Ⅱ

我们说 Database System 指的是 Database 和 DBMS 的结合

  • Database (DB), 指的是数据, 有关联的数据会被存储在表中
  • Database Management System (DBMS) 是管理数据的软件, 例如 MySQL


数据独立性 (Data independence)

应用层只关心数据 以及 relational model, 不关心 DBMS 怎么实现的存储与查找.

无论实施细节如何，DBMS 都为应用层维护着相同的 relational conceptual view


DBMS 的功能

  1. 要维护和应用层的连接
  2. 要管理 data manipulation language (DML)
    • 权限控制: 什么时候执行请求
    • 线程管理: 为请求分配线程执行
  3. 检查命令并生成一个 query plan
  4. 执行 query plan
    • 通过调用存储和 transaction management code
    • 访问数据记录，同时保持各种 constraints and guarantees
  5. 返回结果


应用层和 DBMS 的接口有两种

  • 嵌入式
    • 通过直接使用 API 调用, 例如 SQLite
  • Tiered client-server
    • 应用程序与DBMS的客户通信管理器建立连接. 通过JDBC/ODBC/等等.
    • Connect to some server(s) that connect to the database server

Process Manager

DBMS 会分配线程给每个 “Queries” (例如 SQL command) 来处理


权限管理:

DBMS 的资源有限, 只有再有足够的资源时才会接受新的工作

  • 为了避免 thrashing, 允许 graceful degradation
  • 在使用高峰期，吞吐量仍然很高，延迟增加


Relational Query Processor

如果有足够的资源运行 query

  • Determined by admission control and scheduling components
  • Enough system resources such as memory, CPU cores

就要用 query processor 来进一步处理

  • 检查用户是否有执行 query 的权限
  • 生成 query plan
  • Plan executor 来执行 query plan


Plan Executor

  • Consists of a series of general-purpose “operators” that can be used to execute any query. 例如: select, sort
  • Operators 会调用 transactional storage manager 来访问真实数据


Transactional Storage Manager

处理所有 data accesses and manipulations

Key component for online transaction processing (OLTP)


高效读取/修改数据

  • indexes, tables
  • Buffering and judicious use of storage devices

保持数据完整性(Guarantee data integrity)

  • 防止故障和崩溃时的数据损坏
  • 协调来自多个线程的并发访问


Return Data to Application

“unwinding” the stack of previous activities

Access method returns control to Query Processor

Query processor 生成 data record 把他们放到 buffer 等待 Connection Manager/API interface 处理

Connection Manager/API interface 把数据运给 调用者


物理层级

必须把data从 storage 移动到 memory 以供 CPU 访问


Storage

可以”永久”存储数据

  • hard disk
  • SSD (solid state drive)
  • 非常便宜, 3GB memory 和 2000GB disk 价格差不多
  • 但是访问速度非常慢


Hard Disk

近及年主要的持久化存储格式

由大量的 sectors (扇区) 组成

  • 每个 sector 是 512 byte block
  • 可以看作是 array of sectors

Hard Disk 的访问

  • “Only single sector accesses are atomic” (只在软件层保证, 硬件层不保证)
  • 通常经常需要 multi-sector accesses
    • 例如 文件系统 read/write in 4KB blocks
    • 可能会导致 torn write, 例如在multi-sector write 时断电, 使得有一些 sector 没有成功写入
  • Sequential access 比 random access 更快(例如顺序访问1-100 比乱序更快)

Cache

  • Internal, invisible to software
  • 由设备来保证 “真正的 “持久性


NAND Flash memory

Basic building block: floating gate transistors (cells)

  • Data 存储在 cell 中
  • 每个 cell 中 bit 的数量取决于 voltage levels

Flash memory types

  • Single-level cell (SLC): 1bit/cell (1 threshold voltage)
  • Multi-level cell (MLC): 2 bit/cell (3 threshold voltage)
  • 还有 TLC 和 QLC …

Organized in blocks and pages

  • 只能以 block 或 page level 进行 read / write, 任何修改必须 erase block

Fixed erase cycles

  • SLC best, MLC worse, TLC even worse, QLC worst


Main Memory

DRAM (dynamic random access memory)

需要恒定的电流来保留数据

挥发性 (volatile)：停电时内容会丢失

系统也可能有少量的非易失性RAM（或ROM），通常用于保存基本信息。

  • Bootstrap loader
  • BIOS (Basic input output system)


Memory 和 Storage 相关工具

  • df: Storage usage stats
  • mount: 挂载现有的文件系统以供访问，或显示当前挂载的文件系统
  • iostat: 显示 storage I/O speed
  • free: 显示 memory usage
  • hdparm/sdparm: 显示 I/O device 状态 和 参数


单个处理器

这里指的是物理层面的 thread, 而不是软件层面的


Data Organization

Classic DBMS

Main memory 通常比 storage 小很多

我们需要把数据从 storage 读取到 memory 或 从 memory 写入 storage

明显 Disk/storage 的访问速度决定了整体 DBMS 的性能

I/O on the critical path

现在内存越来越便宜

因此有些 DBMS 直接使用 Main Memory 存储 data

很多 classic DBMS 的约束都不再适用了



数据的结构

Page: 最基本的 I/O 单元 (软件层面的 Page)

通常不会太大: 1KB, 4KB, 8KB 或 16 KB (MySQL default)

Page size 是一个可变的参数, 但是当确定下来后就不太容易改变

和物理层的 Page 不一样


每个 Page 需要

  • a page header
  • a collection of slots, each slot stores a record


Record : fields 的 集合, 用于用户访问

Records can be variable or fixed length

Each record identified by a unique Record ID (RID)

  • RID: $\text{}$ Pair

An RID uniquely identifies a data record


Fixed-Length Record

可以随机访问, 因为可以直接计算出位置


Variable-Length Records

通过分隔符区分每个record

不可以随机访问, 需要扫描来寻找特定的 field


对上面的 variable-length record 进行改进

我们可以把每个 record 的大小存储在最前面, 通过他们来寻找 field 的位置



Page Format: Fixed Length record

由于所有 record 大小都一样, 我们可以像数组那样存储他们

这样就可以进行随机访问

使用 i*sizeof(Record) 来访问第 i+1 个元素

问题是当我们删除一个元素时, 要如何处理?

可以将删除元素后面的元素都向前移动一位

好处: 简单

坏处:

  • 移动元素太耗时
  • 移动 record 时, RID 可能会被改变




基于此, 我们可以进行一些改进

增加一个 bit array 来存储每个record 位置的信息 (1 表示这个 record 正在被使用, 反之是 0)

只需要扫描 bit array 就能知道哪个位置有空

好处:

  • 不需要对 record 进行重新排列
  • RIDs 不会改变

坏处: 需要处理 bit array (但是很快)



Page Format: Variable Length record

record 在这种情况下可以是任意 size (但是通常会比整个 page 小)


slotted page structure

每个 slot entry 有一个 record offset (指向 record 的起始位置) 和一个 record length (用于计算 record 的终止位置)

例如: 上图中 slot directory 的第一个元素指向一个 record (RID=(i, 1)), 这就是 记录 record offset

而它同时知道 record length = 24

这种方式更灵活但也更复杂

删除操作:

  • 将 valid slot 的数量 减一
  • 把 slot entry 的 offset 和 size 设为 invalid
  • 但是我们不能删除 slot entry, 因为 RID 会因此改变
    • 除非删除的 record 是最后一个 slot entry

插入操作:

  • 尝试 reuse 之前创建的 slot (现在已经删除的)
  • 需要扫描 existing slot entries

修改 (扩展) 操作:

扩展会导致现存的空间不够

  • 有两种解决方法:
    1. 可以在 free space 创建一个新的entry, 把值复制过去
    2. 可以在 entry 的最后添加下一段数据的位置 (像一个链表)


Organizing Pages into a Heap File

Heap file: a collection of fixed-size pages (这里叫做 heap 只是因为行为类似与我们从 heap 种 allocate 对象)

这个就是一个 file, 这个 file 中存储多个 Pages

  • Pages 可以随着 file 的增长或缩小进行 allocated 和 decallocate

  • Heap file 中的 Pages 是无序的

    • 只确保我们能找到所有的 pages
  • 支持的操作
    • insert/delete/update a record
    • search for a record, scan for a range of record
  • 核心问题
    • 需要跟踪所有的 page
    • 需要跟踪所有的 free space


Heap File: 双向链表

经典 allocate 的算法



访问数据

DB record 必须先从硬盘读取到内存在进行访问

并且在需要的时候从 main memory 中 remove (evict) page

需要使用 Buffer Pool 来进行数据访问


Buffer Frame Replacement Policies

事实: Buffer pool 的大小远远小于 data size

我们可能需要释放 buffer pool 中的一些 page 来读取新的 page

这个操作对性能有极大的影响

  • 取决与访问的方式 (workload)


主要的几种方式

  • Random
  • First-in-first-out (FIFO)
  • Least Recently Used (LRU)
  • Clock Replacement
  • Most Recently Used (MRU)


Random

随机释放可能在实际使用时的性能超过 LRU

因为 LRU 还需要使用锁以及一大堆操作来维护, 而 Random 没有这些需求


LRU

Clock Replacement


为什么我们不使用系统的 cache 而是实现自己的 Buff Pool?

  1. DBMS 知道 workload
  2. 我们可以预测 prefetching, customized replacement 的 pattern
  3. 有时我们需要强制将缓存中的内容写入硬盘
  4. 操作系统间存在差异, OS file 通常不能跨磁盘


Indexing


B+ - Tree

在数据库中, 当讨论 B-Tree 时, 通常说的是 B+Tree

B+Tree 只有 leaf node 存储数据, 而 B-Tree 是每个节点都存储数据



B-Tree 插入

遍历树找到对应 leaf 节点 $L$

把 data entry 放入 $L$

  • 如果 $L$ 有足够的空间, 结束 (每个 node 有 max size limit, 一般来说是一个 page)
  • 如果 $L$ 没有足够空间, 则把 $L$ 拆分为 $L_1,L_2$
    • Redistribute entries evenly
    • 向父节点插入 middle key (新的 separator key)
      • 可能会引发进一步的拆分, 甚至会传播到 root node



B-Tree 删除

遍历树找到对应 leaf 节点 $L$

删除 entry

  • 如果 $L$ 还有一半以上的 数据 (半满), 结束
  • 反之 需要 redistribute 或 merge
    • Redistribute: 从 sibling 借一条数据
    • Merge: 和 sibling 组合成一个新的 node, 删除 父节点中的 key pointer
      • 如果 父节点的数据因此小于一半, 继续向上传播
      • 可能传播到 root 节点并且缩小 tree 的高度


S-latch 是读锁, X-latch 是写锁


Latch Coupling (Crabbing)

在访问 单个node 之前 就获取需要的 S/X latches

如果你确定任何在子节点的操作不会影响父亲节点

  • update 操作 不会 split/merge node
  • 叶子节点 not full (可以 insert), not less than half full (可以delete)

当 子节点 成功 latched 后, 再 Unlatch 父节点


Search 一个 single key (point query)

  • 在 root 节点请求 S-latching
  • 在 child node 请求 S-latch
  • 释放 parent node 的 S-latch

Skip List

当我们进行 traversal 时

要用 stack/array 来存储路径

这对 reverse scan, insert, delete 非常有用

  • Reverse scan: find start key, then move backwards using the stack
  • Insert/delete: insert at the lowest level, then build up towers

Concurrency


解决思路:

这种方式是 lock free 算法的一个常用的方式

使用 logically delete, 而不是真的 delete

我们可以使用一个bit来表示这个node是否真的被删除

删除操作 (设置 delete bit 为 1):

  • 仅仅是逻辑上删除
  • 依然可以被遍历到 (但是会被 filtered out)
  • 只有当真的没有线程使用时才会回收 (物理删除)

插入操作:

  • 只有在 predecessor 的 delete bit 为 0 时插入

  • 当 predecessor 被 logically deleted 时会无法插入


优化

对于每个 “tower”, 只存储一个值, 可以使用多个pointer代表多层

减少 cache misses, 提高 space efficiency


我们还可以把相邻的 block 合并起来, 这样也减少的指针的使用

但我们无法预测每个 block 开辟空间的大小, 因此有时候我们需要预先开辟空间, 当我们再插入元素是可以使用这些空间


在此之上还可以减少 random 函数的使用次数

只调用一次random确定层数

或者使用 random() % max_level + 1 来确定层数


Skip Lists 对比 B+-Tree

  • 二者都保存了数据的 order
  • 二者都被广泛使用
    • B-trees 一般在所有 DBMS 中都使用
    • Skip lists 也被 一些DBMS 中使用, 同时在 key-value stores (KVS) 中 也很受欢迎
      • Redis (KVS), LevelDB (KVS), SingleStore (DBMS), RocksDB (DBMS), WiredTiger (DBMS)

B+-trees

  • 在存储上更为优化, 比 skip lists 更加 cache efficient
  • 很难 latch-free
  • 更容易支持 forward and reverse range scan

Skip lists

  • 对 disk/storage 不友好, not optimized for locality
  • latch-free 更容易
  • reverse range scan 稍微复杂一点


什么是 Lock/Latch Free 算法

内存管理可能会很 tricky

  • 必须要确保 unlink node 不被 recycle 直到没有线程使用它们
    • 不然会引发 Segfault 或 访问到垃圾数据
  • 一些解决方案
    • Hazard pointers
    • Epoch-based reclamation
    • Reference count

Lock-free: ensures overall progress

  • 使用锁是确保只有一个线程往前执行 (critical section), 而其他线程在做一些其他的事情
  • 而在 Lock-free 中没有任何 critical section, 所有线程能访问所有变量的所有中间状态



Hash Table

这里我们更注重 hash table 在 DBMS 中是如何 indexing 的

Hash table 是一组 “buckets”, 每个 “bucket” 存储 keys 以及 associated payload (相关的有效载荷)

我们假设有一个比较不错的 hash function

  • hash function 可以 把一个 key 映射 为 一个 bucket index
    • 可能会有 collision, 例如对于不同 k1, k2, hash(k1) == hash(k2)
    • 因此每个 bucket 会存储多个 data entries
  • 优点: point queries 的速度非常快

  • 通常不支持 range scan

    • hash function 通常不会保持 order
    • 很有可能 k1 < k2 但 hash(k1) > hash(k2)

  • Technique:

    • Static hashing
    • Dynamic hashing: extendible hashing and linear hashing



Static Hashing

bucket 的数量是固定的

每个 bucket 包含 一个或多个 page

每个 bucket page 的 capacity 都是固定的

  • 如果一个 page 没有 space: create overflow pages chained together per bucket


Hash function 必须 均匀地把数据分布到不同的 bucket 中

主要的问题: bucket 的数量是固定的

  • Chain of overflow pages 可能会太长
    • 影响 search performance
  • 当 data size 显著减小时, 会浪费空间
    • Empty buckets
  • 当我们 增加/减少 bucket 数量时需要 rehashing 需要拷贝太多数据


Extendible Hashing

“All problems in computer science can be solved by another level of indirection”

“But that usually will create another problem.”

— David Wheeler


“…except for the problem of too many layers of indirection.”

— Kevlin Henney

Use a level of indirection to easily add more buckets

添加一个 directory of pointers (bucket id) 来指向 buckets

  • 比实际的 bucket page 小很多, resize/expand 的花销更小
  • 当 directory 满了就 双倍扩建
    • Only split buckets that are full
    • No overflow pages

我们可以假设 bucket 是 fixed size, 或者说只有 1 page

我们假设 H(30) mod 4 = 1 , H(30) mod 8 = 5

我们只对 correspond bucket 进行 re-indexing

如果 local depth != global depth, 说明 directory 中还有一个 correspond pointer, 就是说 依然可以存储元素, 不需要 double directory
